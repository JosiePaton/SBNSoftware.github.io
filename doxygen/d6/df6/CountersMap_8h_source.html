95  /**
96  * @brief Map storing counters in a compact way
97  * @param KEY the type of the key of the counters map
98  * @param COUNTER the type of a basic counter (can be signed or unsigned)
99  * @param BLOCKSIZE the number of counters in a cluster
100  * @param ALLOC allocator for the underlying STL map
101  * @param SUBCOUNTERS split each counter in subcounters (not implemented yet)
102  *
103  * This class is designed for the need of a vast number of counters with
104  * a integral numerical key, when the counter keys are usually clustered.
105  * This can be more or less efficient than a straight counters STL map
106  * according to how often the counters are clustered (if that does not happen
107  * often, CountersMap can have considerable memory overhead).
108  *
109  * Counters are allocated in contiguous blocks of BLOCKSIZE counters.
110  * The selling point here is that a map node has some overhead (typically
111  * at least 3 pointers) and dynamically allocating it costs a lot (40 bytes
112  * have been observed). If you need a counter with a range of 100 (1 byte),
113  * that's far from optimal in many aspects (memory allocation also takes
114  * time).
115  *
116  * The requirement of the key to be numerical is so that the concept of
117  * "next counter" is well defined and we can store contiguous counters
118  * in a fixed structure.
119  *
120  * <h3>Subcounters</h3>
121  * The idea behind subcounters is that you migt want to split a counter into
122  * subcounters to save memory if the maximum counter value is smaller than
123  * the range of the counter type.
124  * <strong>The implementation is delayed</strong> since in the end the same
125  * effect can be achieved by using a small counter type (e.g. signed char),
126  * unless the range is smaller that 16 (or 4, or 2), in which case the
127  * character can be split into bits. That will cause some overhead for
128  * increment and decrement instructions.
129  */
